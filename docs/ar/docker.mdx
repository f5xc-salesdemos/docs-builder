يُوزَّع المُنشئ كصورة Docker. تقوم مستودعات المحتوى بتشغيله لإنتاج HTML ثابت دون الحاجة لتثبيت Node.js أو Astro محلياً.

## Dockerfile

يستخدم `docker/Dockerfile` بناءً متعدد المراحل على `node:24-alpine`:

```dockerfile
FROM node:24-alpine AS builder
WORKDIR /app

# Install deps (cached layer)
COPY package*.json ./
RUN npm ci

# Copy framework code
COPY . .

# Remove placeholder content
RUN rm -rf src/content/docs/*

COPY docker/entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

### استراتيجية الطبقات

| الطبقة | المحتوى | سبب التخزين المؤقت المنفصل |
|--------|---------|---------------------------|
| 1 | `COPY package*.json` + `npm ci` | تتغير التبعيات بشكل نادر؛ تخزين هذه الطبقة مؤقتاً يتجنب إعادة التنزيل في كل عملية بناء |
| 2 | `COPY . .` | الكود المصدري لإطار العمل |
| 3 | `rm -rf src/content/docs/*` | يُنظف أي محتوى مؤقت بحيث يظهر فقط المحتوى المُحقن |
| 4 | `COPY entrypoint.sh` | سكربت تنسيق عملية البناء |

## نقطة الدخول

يعمل `docker/entrypoint.sh` في كل مرة يبدأ فيها الحاوية. يقوم بتنفيذ هذه الخطوات بالترتيب:

### الخطوة 1: تحديث التبعيات

```sh
npm install
npm update
```

يضمن أن حزمة القالب وجميع التبعيات في أحدث إصداراتها المتوافقة.

### الخطوة 2: نسخ إعدادات القالب

```sh
cp /app/node_modules/@f5xc-salesdemos/docs-theme/astro.config.mjs /app/astro.config.mjs
cp /app/node_modules/@f5xc-salesdemos/docs-theme/src/content.config.ts /app/src/content.config.ts
```

ينسخ `astro.config.mjs` و `content.config.ts` من حزمة القالب. القالب هو المصدر الوحيد للحقيقة لإعدادات Astro — انظر [الهندسة المعمارية](../architecture/) لتصميم نظام القالب.

### الخطوة 3: حقن المحتوى

```sh
if [ -d "$CONTENT_DIR" ]; then
  cp -r "$CONTENT_DIR"/* /app/src/content/docs/
else
  echo "ERROR: No content found at $CONTENT_DIR"
  exit 1
fi
```

ينسخ المحتوى المُركَّب إلى مجلد مجموعة محتوى Astro. يخرج مع خطأ إذا كان مجلد المحتوى مفقوداً.

### الخطوة 4: استخراج العنوان

```sh
if [ -z "$DOCS_TITLE" ] && [ -f /app/src/content/docs/index.mdx ]; then
  DOCS_TITLE=$(grep -m1 '^title:' /app/src/content/docs/index.mdx | sed 's/title: *["]*//;s/["]*$//')
  export DOCS_TITLE
fi
```

إذا لم يتم تعيين `DOCS_TITLE` عبر البيئة، يستخرجه من حقل `title:` في frontmatter الخاص بـ `index.mdx`.

### الخطوة 5: استخراج المسار الأساسي

```sh
if [ -z "$DOCS_BASE" ] && [ -n "$GITHUB_REPOSITORY" ]; then
  DOCS_BASE="/${GITHUB_REPOSITORY#*/}"
  export DOCS_BASE
fi
```

يشتق المسار الأساسي للموقع من اسم مستودع GitHub (مثلاً، `owner/my-docs` يصبح `/my-docs`).

### الخطوة 6: البناء

```sh
npm run build
```

يُشغّل `astro build`، وينتج مخرجات ثابتة في `/app/dist/`.

### الخطوة 7: نسخ المخرجات

```sh
if [ -d "$OUTPUT_DIR" ]; then
  cp -r /app/dist/* "$OUTPUT_DIR"/
fi
```

ينسخ الموقع المبني إلى نقطة تركيب المخرجات.

## متغيرات البيئة

| المتغير | القيمة الافتراضية | الوصف |
|---------|-------------------|-------|
| `CONTENT_DIR` | `/content/docs` | مسار مجلد المحتوى المُركَّب |
| `OUTPUT_DIR` | `/output` | المسار الذي يُنسخ إليه الموقع المبني |
| `DOCS_TITLE` | *(يُستخرج من index.mdx)* | عنوان الموقع المُمرر إلى إعدادات Astro |
| `DOCS_DESCRIPTION` | *(يُستخرج من index.mdx)* | وصف الموقع من frontmatter |
| `DOCS_BASE` | *(يُشتق من `GITHUB_REPOSITORY`)* | المسار الأساسي للموقع (مثلاً، `/my-docs`) |
| `DOCS_SITE` | *(يُشتق من `GITHUB_REPOSITORY_OWNER`)* | عنوان URL للموقع المُمرر إلى إعدادات Astro (مثلاً، `https://<owner>.github.io`) |
| `GITHUB_REPOSITORY` | *(يُعيّنه GitHub Actions)* | سلسلة `owner/repo` المستخدمة لاشتقاق `DOCS_BASE` |
| `GENERATE_PDF` | `false` | تفعيل إنشاء PDF بعد البناء |
| `PDF_FILENAME` | `docs` | اسم ملف الإخراج لملف PDF المُنشأ |
| `LLMS_OPTIONAL_LINKS` | `[]` | مصفوفة JSON لعناوين URL للمواقع الفرعية لقسم Optional في `llms.txt` |

## الاستخدام المحلي

لبناء الوثائق محلياً باستخدام Docker:

```sh
docker run --rm \
  -v "$(pwd)/docs:/content/docs" \
  -v "$(pwd)/output:/output" \
  ghcr.io/f5xc-salesdemos/docs-builder:latest
```

يُركّب هذا مجلد `docs/` المحلي كمحتوى ويكتب الموقع المبني إلى `output/`.

لمؤلفي المحتوى الذين يريدون سير عمل خادم تطوير مباشر، انظر [المعاينة المحلية لمؤلفي المحتوى](../content-authors/).

## للمطورين: خادم التطوير المحلي

عند العمل على المُنشئ نفسه (المكونات، الإضافات، تكامل القالب)، يمكنك تشغيل خادم تطوير Astro مباشرة دون Docker:

```sh
npm ci
cp node_modules/@f5xc-salesdemos/docs-theme/astro.config.mjs astro.config.mjs
cp node_modules/@f5xc-salesdemos/docs-theme/src/content.config.ts src/content.config.ts
# Copy some test content into the content collection
cp -r /path/to/test-content/* src/content/docs/
DOCS_TITLE="Dev Test" npx astro dev --host
```

افتح `http://localhost:4321`. يعمل الاستبدال السريع للوحدات (HMR) لتغييرات المكونات والإضافات.

:::note
`astro.config.mjs` و `content.config.ts` يأتيان من حزمة القالب ولا يخضعان للتحكم في الإصدارات في هذا المستودع. يجب نسخهما من `node_modules` قبل تشغيل خادم التطوير. تتعامل نقطة الدخول مع هذا تلقائياً في عمليات بناء Docker.
:::

:::note
يتم تفريغ `src/content/docs/` في وقت بناء الصورة ويتم تجاهله في git. ضع محتوى الاختبار هناك يدوياً — لن يتم تأكيده في المستودع.
:::

## سجل الصور

تُنشر الصورة في GitHub Container Registry:

```
ghcr.io/f5xc-salesdemos/docs-builder
```

يتم دفع وسمين في كل عملية بناء:

| الوسم | الوصف |
|-------|-------|
| `latest` | أحدث بناء من فرع `main` |
| `<sha>` | معرّف SHA الكامل لعملية الإيداع في Git للبناءات القابلة للإعادة |

انظر [CI/CD والحوكمة](../ci-cd/) لسير العمل الذي يبني وينشر الصورة.
