يستخدم المستودع أربعة مسارات عمل GitHub Actions ونظام قوالب مركزي للحوكمة.

## مسارات العمل

| مسار العمل | الملف | المحفز | الغرض |
|----------|------|---------|---------|
| نشر GitHub Pages | `github-pages-deploy.yml` | الدفع إلى `main` (`docs/**`)، التشغيل اليدوي | بناء ونشر موقع التوثيق على GitHub Pages |
| بناء ونشر صورة Docker | `build-image.yml` | الدفع إلى `main` (`docker/**`، `package*.json`)، مهمة مجدولة يومية، `repository_dispatch` | بناء صورة Docker، دفعها إلى GHCR، وإرسال أوامر إعادة البناء للمستودعات التابعة |
| طلب ربط مشكلة | `require-linked-issue.yml` | أحداث طلبات السحب | حظر طلبات السحب التي لا تشير إلى مشكلة GitHub |
| فرض إعدادات المستودع | `enforce-repo-settings.yml` | كل 6 ساعات، الدفع إلى ملف تكوين الإعدادات، التشغيل اليدوي | تطبيق إعدادات المستودع الموحدة من القالب |

### نشر GitHub Pages

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'docs/**'
  workflow_dispatch:
```

يفوض إلى مسار عمل قابل لإعادة الاستخدام في مستودع القالب:

```yaml
jobs:
  docs:
    uses: f5xc-salesdemos/docs-control/.github/workflows/github-pages-deploy.yml@main
```

الصلاحيات: `contents: read`، `packages: read`، `pages: write`، `id-token: write`. يستخدم مجموعة تزامن `pages` مع `cancel-in-progress: true` لتجنب عمليات النشر القديمة.

يتم التشغيل فقط عند الدفع عندما تتغير الملفات الموجودة ضمن `docs/`. يمكن أيضاً تشغيله يدوياً عبر `workflow_dispatch`، وهذه هي الطريقة التي تقوم بها مهمة الإرسال في `build-image.yml` بتشغيل إعادة بناء المستودعات التابعة.

### بناء ونشر صورة Docker

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'docker/**'
      - 'package.json'
      - 'package-lock.json'
  schedule:
    - cron: '0 6 * * *'
  repository_dispatch:
    types: [rebuild-image]
```

الخطوات:
1. استخراج الكود
2. تسجيل الدخول إلى `ghcr.io` باستخدام `docker/login-action`
3. البناء والدفع باستخدام `docker/build-push-action` مع السياق `.` والملف `docker/Dockerfile`
4. الوسوم: `ghcr.io/<owner>/<repo>:latest` و `ghcr.io/<owner>/<repo>:<sha>`

بعد بناء ناجح، تقوم مهمة **الإرسال** بتشغيل `github-pages-deploy.yml` عبر `workflow_dispatch` في كل مستودع تابع مدرج في ملف تكوين `downstream-repos.json` الخاص بمستودع القالب. يضمن هذا أن جميع مستودعات المحتوى تعيد بناء توثيقها باستخدام صورة البناء المحدثة.

تضمن المهمة المجدولة اليومية أن تبقى الصورة محدثة حتى بدون تغييرات في الكود (تلتقط تحديثات التبعيات). يسمح حدث `repository_dispatch` للأنظمة الخارجية بتشغيل إعادة البناء.

يتم التشغيل فقط عند الدفع عندما تتغير ملفات `docker/` أو `package*.json`. لا تؤدي التغييرات المقتصرة على التوثيق إلى تشغيل إعادة بناء الصورة.

### طلب ربط مشكلة

```yaml
on:
  pull_request_target:
    types: [opened, edited, reopened, synchronize]
```

يستخدم `nearform-actions/github-action-check-linked-issues@v1` لفرض أن كل طلب سحب يشير إلى مشكلة GitHub (مثل `Closes #42` في الوصف). يتم استثناء طلبات سحب Dependabot عبر:

```yaml
exclude-branches: "dependabot/**"
```

تخبر رسالة مخصصة المساهمين بالتنسيق المتوقع إذا فشل الفحص. راجع [CONTRIBUTING.md](../CONTRIBUTING.md) لمعرفة سير عمل المساهمة الكامل.

### فرض إعدادات المستودع

```yaml
on:
  schedule:
    - cron: '0 */6 * * *'
  push:
    branches: [main]
    paths:
      - '.github/config/repo-settings.json'
  workflow_dispatch:
```

يفوض إلى مسار عمل قابل لإعادة الاستخدام في مستودع القالب:

```yaml
jobs:
  enforce:
    uses: f5xc-salesdemos/docs-control/.github/workflows/enforce-repo-settings.yml@main
    secrets:
      repo-admin-token: ${{ secrets.REPO_ADMIN_TOKEN }}
```

يعمل كل 6 ساعات وعند إجراء تغييرات على ملف تكوين الإعدادات. يطبق قواعد حماية الفروع، وإعدادات الدمج، وتكوينات المستودع الأخرى من `.github/config/repo-settings.json`.

## نموذج الحوكمة

### القالب المركزي

مستودع `f5xc-salesdemos/docs-control` هو المصدر الوحيد للحقيقة لكل من:

- تعريفات مسارات العمل القابلة لإعادة الاستخدام (نشر التوثيق، فرض إعدادات المستودع)
- الملفات المُدارة المتزامنة عبر جميع المستودعات في المنظمة
- التكوينات القياسية وقواعد حماية الفروع

يرث هذا المُنشئ وجميع مستودعات المحتوى سلوك CI/CD الخاص بها من القالب.

### مزامنة الملفات المُدارة

يمكن لمستودع القالب دفع الملفات المُدارة (مثل تعريفات مسارات العمل وملفات التكوين) إلى المستودعات التابعة. يحافظ هذا على محاذاة جميع المستودعات دون تحديثات يدوية. تعمل المزامنة عبر مسار عمل منفصل في مستودع القالب.

### حماية الفروع

يطبق مسار عمل فرض إعدادات المستودع قواعد حماية الفروع من القالب، بما في ذلك:

- فحوصات الحالة المطلوبة قبل الدمج
- فحص ربط المشكلة المطلوب
- تفضيل دمج السكواش (Squash merge)
- الحذف التلقائي لفروع الرأس بعد الدمج

## الأسرار

| السر | المصدر | الغرض |
|--------|--------|---------|
| `GITHUB_TOKEN` | تلقائي (GitHub) | يُستخدم بواسطة معظم مسارات العمل لاستخراج الكود ونشر الحزم واستدعاءات API |
| `REPO_ADMIN_TOKEN` | يدوي (سر المستودع) | مطلوب بواسطة مسار عمل فرض إعدادات المستودع ومهمة الإرسال لتعديل حماية الفروع وإعدادات المستودع وتشغيل مسارات العمل التابعة (يحتاج صلاحية المسؤول) |
