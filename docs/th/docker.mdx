ตัว builder จัดส่งในรูปแบบ Docker image โดย content repo จะรันเพื่อสร้าง static HTML โดยไม่ต้องติดตั้ง Node.js หรือ Astro ในเครื่อง

## Dockerfile

`docker/Dockerfile` ใช้การ build แบบหลายขั้นตอนบน `node:24-alpine`:

```dockerfile
FROM node:24-alpine AS builder
WORKDIR /app

# Install deps (cached layer)
COPY package*.json ./
RUN npm ci

# Copy framework code
COPY . .

# Remove placeholder content
RUN rm -rf src/content/docs/*

COPY docker/entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

### กลยุทธ์เลเยอร์

| เลเยอร์ | สิ่งที่ทำ | เหตุผลที่แคชแยก |
|---------|----------|----------------|
| 1 | `COPY package*.json` + `npm ci` | Dependencies เปลี่ยนแปลงไม่บ่อย; การแคชเลเยอร์นี้ช่วยหลีกเลี่ยงการดาวน์โหลดซ้ำในทุกครั้งที่ build |
| 2 | `COPY . .` | ซอร์สโค้ดของ framework |
| 3 | `rm -rf src/content/docs/*` | ล้างเนื้อหา placeholder ออกเพื่อให้แสดงเฉพาะเนื้อหาที่ถูก inject เข้ามา |
| 4 | `COPY entrypoint.sh` | สคริปต์จัดการการ build |

## Entrypoint

`docker/entrypoint.sh` จะรันทุกครั้งที่ container เริ่มทำงาน โดยดำเนินการตามขั้นตอนต่อไปนี้ตามลำดับ:

### ขั้นตอนที่ 1: อัปเดต Dependencies

```sh
npm install
npm update
```

ตรวจสอบให้แน่ใจว่า theme package และ dependencies ทั้งหมดเป็นเวอร์ชันล่าสุดที่เข้ากันได้

### ขั้นตอนที่ 2: คัดลอก Theme Config

```sh
cp /app/node_modules/@f5xc-salesdemos/docs-theme/astro.config.mjs /app/astro.config.mjs
cp /app/node_modules/@f5xc-salesdemos/docs-theme/src/content.config.ts /app/src/content.config.ts
```

คัดลอก `astro.config.mjs` และ `content.config.ts` จาก theme package โดย theme เป็นแหล่งข้อมูลเดียวที่เชื่อถือได้สำหรับการตั้งค่า Astro — ดู [สถาปัตยกรรม](../architecture/) สำหรับการออกแบบระบบ theme

### ขั้นตอนที่ 3: Inject เนื้อหา

```sh
if [ -d "$CONTENT_DIR" ]; then
  cp -r "$CONTENT_DIR"/* /app/src/content/docs/
else
  echo "ERROR: No content found at $CONTENT_DIR"
  exit 1
fi
```

คัดลอกเนื้อหาที่ mount เข้ามาไปยังไดเรกทอรี content collection ของ Astro จะออกพร้อม error หากไม่พบไดเรกทอรีเนื้อหา

### ขั้นตอนที่ 4: ดึงชื่อเรื่อง

```sh
if [ -z "$DOCS_TITLE" ] && [ -f /app/src/content/docs/index.mdx ]; then
  DOCS_TITLE=$(grep -m1 '^title:' /app/src/content/docs/index.mdx | sed 's/title: *["]*//;s/["]*$//')
  export DOCS_TITLE
fi
```

หาก `DOCS_TITLE` ไม่ได้ตั้งค่าผ่านตัวแปรสภาพแวดล้อม จะดึงจากฟิลด์ frontmatter `title:` ของ `index.mdx`

### ขั้นตอนที่ 5: ดึง Base Path

```sh
if [ -z "$DOCS_BASE" ] && [ -n "$GITHUB_REPOSITORY" ]; then
  DOCS_BASE="/${GITHUB_REPOSITORY#*/}"
  export DOCS_BASE
fi
```

คำนวณ base path ของเว็บไซต์จากชื่อ GitHub repository (เช่น `owner/my-docs` จะกลายเป็น `/my-docs`)

### ขั้นตอนที่ 6: Build

```sh
npm run build
```

รัน `astro build` สร้างผลลัพธ์แบบ static ใน `/app/dist/`

### ขั้นตอนที่ 7: คัดลอกผลลัพธ์

```sh
if [ -d "$OUTPUT_DIR" ]; then
  cp -r /app/dist/* "$OUTPUT_DIR"/
fi
```

คัดลอกเว็บไซต์ที่ build แล้วไปยังจุด mount ของผลลัพธ์

## ตัวแปรสภาพแวดล้อม

| ตัวแปร | ค่าเริ่มต้น | คำอธิบาย |
|--------|------------|----------|
| `CONTENT_DIR` | `/content/docs` | พาธไปยังไดเรกทอรีเนื้อหาที่ mount |
| `OUTPUT_DIR` | `/output` | พาธที่เว็บไซต์ที่ build แล้วจะถูกคัดลอกไป |
| `DOCS_TITLE` | *(ดึงจาก index.mdx)* | ชื่อเว็บไซต์ที่ส่งไปยัง Astro config |
| `DOCS_DESCRIPTION` | *(ดึงจาก index.mdx)* | คำอธิบายเว็บไซต์จาก frontmatter |
| `DOCS_BASE` | *(คำนวณจาก `GITHUB_REPOSITORY`)* | Base path สำหรับเว็บไซต์ (เช่น `/my-docs`) |
| `DOCS_SITE` | *(คำนวณจาก `GITHUB_REPOSITORY_OWNER`)* | URL ของเว็บไซต์ที่ส่งไปยัง Astro config (เช่น `https://<owner>.github.io`) |
| `GITHUB_REPOSITORY` | *(ตั้งค่าโดย GitHub Actions)* | สตริง `owner/repo` ที่ใช้คำนวณ `DOCS_BASE` |
| `GENERATE_PDF` | `false` | เปิดใช้งานการสร้าง PDF หลัง build |
| `PDF_FILENAME` | `docs` | ชื่อไฟล์ผลลัพธ์สำหรับ PDF ที่สร้าง |
| `LLMS_OPTIONAL_LINKS` | `[]` | JSON array ของ URL เว็บไซต์ย่อยสำหรับส่วน Optional ของ `llms.txt` |

## การใช้งานในเครื่อง

เพื่อ build เอกสารในเครื่องโดยใช้ Docker:

```sh
docker run --rm \
  -v "$(pwd)/docs:/content/docs" \
  -v "$(pwd)/output:/output" \
  ghcr.io/f5xc-salesdemos/docs-builder:latest
```

คำสั่งนี้จะ mount ไดเรกทอรี `docs/` ในเครื่องเป็นเนื้อหาและเขียนเว็บไซต์ที่ build แล้วไปยัง `output/`

สำหรับผู้เขียนเนื้อหาที่ต้องการเวิร์กโฟลว์ dev server แบบ live ดู [การพรีวิวในเครื่องสำหรับผู้เขียนเนื้อหา](../content-authors/)

## นักพัฒนา: Dev Server ในเครื่อง

เมื่อทำงานกับตัว builder เอง (components, plugins, การรวม theme) คุณสามารถรัน dev server ของ Astro โดยตรงโดยไม่ต้องใช้ Docker:

```sh
npm ci
cp node_modules/@f5xc-salesdemos/docs-theme/astro.config.mjs astro.config.mjs
cp node_modules/@f5xc-salesdemos/docs-theme/src/content.config.ts src/content.config.ts
# Copy some test content into the content collection
cp -r /path/to/test-content/* src/content/docs/
DOCS_TITLE="Dev Test" npx astro dev --host
```

เปิด `http://localhost:4321` Hot module replacement (HMR) ทำงานได้กับการเปลี่ยนแปลง component และ plugin

:::note
`astro.config.mjs` และ `content.config.ts` มาจาก theme package และไม่ได้อยู่ในระบบ version control ของ repo นี้ คุณต้องคัดลอกจาก `node_modules` ก่อนรัน dev server โดย entrypoint จะจัดการเรื่องนี้โดยอัตโนมัติใน Docker builds
:::

:::note
`src/content/docs/` จะถูกล้างในขั้นตอน build ของ image และถูก gitignore ให้วางเนื้อหาทดสอบที่นั่นด้วยตนเอง — มันจะไม่ถูก commit
:::

## Image Registry

Image ถูกเผยแพร่ไปยัง GitHub Container Registry:

```
ghcr.io/f5xc-salesdemos/docs-builder
```

มีสอง tag ที่ถูก push ในทุกครั้งที่ build:

| Tag | คำอธิบาย |
|-----|----------|
| `latest` | Build ล่าสุดจาก `main` |
| `<sha>` | Git commit SHA เต็มสำหรับการ build ที่ทำซ้ำได้ |

ดู [CI/CD และการกำกับดูแล](../ci-cd/) สำหรับ workflow ที่ build และเผยแพร่ image
